## Library Build Order

Library packages must be built before CMS packages can start. Order matters due to dependencies:

1. `packages/draft-renderer` → `make build`
2. `packages/draft-editor` → `make build`
3. `packages/core` → `make build`

Each `make build` runs `babel src --out-dir lib` (+ `tsc` for `core`).

## Keystone Express Middleware Ordering

**Critical**: In Keystone 6's `createExpressServer.ts`, the order is:

1. CORS middleware
2. Health check route
3. **`extendExpressApp(app, context)`** ← user middleware goes here
4. `extendHttpServer`
5. Storage static routes (`express.static` for each `storage[].serverRoute`)
6. Apollo/GraphQL middleware

This means middleware registered in `extendExpressApp` runs **before** Keystone's storage static file serving. Auth guards for static paths must be registered here.

## Express Mini-Apps Pattern

Mini-apps are Express Routers mounted in `keystone.ts`'s `extendExpressApp`. They receive `keystoneContext` and use `keystoneContext.withRequest(req, res)` to access the session.

### Session check pattern

```js
const authenticationMw = async (req, res, next) => {
  const context = await keystoneContext.withRequest(req, res)
  if (context?.session?.data) {
    return next()
  }
  res.status(401).send('Unauthorized')  // or res.redirect('/signin')
}
```

### Registered mini-apps (mirrortv)

| Mini-app | File | Condition | Protected paths |
| --- | --- | --- | --- |
| Image auth | `express-mini-apps/images/app.js` | Always | `/images/*` |
| Preview proxy | `express-mini-apps/preview/app.js` | `ACL.CMS` only | `/preview-server/*` |

## mirrortv: Image Auth Middleware

**Feature**: `/images/*` requires a valid Keystone session (logged-in user).

**File**: `packages/mirrortv/express-mini-apps/images/app.js`

**How it works**:
- `createImageAuthMiniApp({ keystoneContext })` returns an Express Router
- Router intercepts `GET /images/*` before Keystone's `express.static` middleware
- Unauthenticated requests → `401 Unauthorized`
- Authenticated requests → `next()` (falls through to Keystone's static file serving)

**Paths not yet protected** (reserved for future): `/files/*`, `/video-files/*`

**URL format for images**: `http://<host>/images/<file_id>.<file_extension>`

## Access Control Strategies

Controlled by `ACCESS_CONTROL_STRATEGY` env var:

| Strategy | Behavior |
| --- | --- |
| `cms` | Standard role-based access. Preview mini-app enabled. |
| `gql` | Public read (for frontend APIs); published items only |
| `preview` | All items readable (for preview servers) |
| `restricted` | Blocks User list queries/updates/deletes by default |

Roles: `admin`, `moderator`, `editor`, `contributor`

## Session

All CMS apps use `statelessSessions` (JWT-style) from `@keystone-6/core/session`.

```ts
const session = statelessSessions({
  secret: SESSION_SECRET,
  maxAge: SESSION_MAX_AGE  // default: 86400s (1 day)
})
```

Session data: `'id name role'`

## Storage (local)

Each CMS app may configure local storage in `keystone.ts`:

```ts
storage: {
  images: {
    kind: 'local',
    type: 'image',
    storagePath: 'public/images',
    serverRoute: { path: '/images' },
    generateUrl: (path) => `/images${path}`,
  }
}
```

Keystone auto-creates `express.static` middleware for each `serverRoute`. Auth guards must be added in `extendExpressApp` (runs first).
